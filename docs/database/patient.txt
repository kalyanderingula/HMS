# Enterprise Hospital Management System (HMS)

# Patient Management Module Documentation

---

# 1. Module Overview

The Patient Management module is the foundation of the Hospital Management System (HMS). It manages all patient-related information including:

* Patient registration
* Demographics
* Identity management
* Contact information
* Insurance
* Medical history
* Allergies
* Documents
* Consent management
* Portal accounts
* Audit history

This module is connected with:

* Appointments
* Admissions
* Billing
* Pharmacy
* Laboratory
* Radiology
* Insurance
* EMR/EHR

---

# 2. Database Design Principles

This schema follows enterprise-level standards:

* UUID-based primary keys
* Normalized relational design
* Foreign key constraints
* Audit tracking
* Soft delete support
* Multi-tenant ready architecture
* HIPAA/GDPR-friendly structure
* Scalable indexing strategy

# 4. Core Patient Tables

---

# TABLE: patients

## Purpose

This is the root table of the Patient Management module.

## Why This Table Exists

Stores the core identity and demographic information for each patient.

## Relationships

* One patient can have many contacts.
* One patient can have many addresses.
* One patient can have many insurance records.
* One patient can have many allergies.
* One patient can have many documents.

---

| Column Name       | Data Type    | Description                      |
| ----------------- | ------------ | -------------------------------- |
| patient_id        | UUID         | Primary key                      |
| tenant_id         | UUID         | Multi-tenant identifier          |
| patient_code      | VARCHAR(50)  | Unique patient code              |
| mrn               | VARCHAR(50)  | Medical Record Number            |
| first_name        | VARCHAR(100) | First name                       |
| middle_name       | VARCHAR(100) | Middle name                      |
| last_name         | VARCHAR(100) | Last name                        |
| date_of_birth     | DATE         | Birth date                       |
| gender_id         | BIGINT       | FK to genders                    |
| blood_group_id    | BIGINT       | FK to blood_groups               |
| marital_status_id | BIGINT       | FK to marital_statuses           |
| status_id         | BIGINT       | FK to patient_statuses           |
| deceased_flag     | BOOLEAN      | Indicates if patient is deceased |
| deceased_date     | TIMESTAMP    | Date of death                    |
| created_at        | TIMESTAMP    | Record creation timestamp        |
| updated_at        | TIMESTAMP    | Record update timestamp          |
| created_by        | UUID         | User who created record          |
| updated_by        | UUID         | User who updated record          |
| deleted_at        | TIMESTAMP    | Soft delete timestamp            |

# 6. Enterprise-Level Features

## Soft Delete

Uses deleted_at column instead of permanent deletion.

## Audit Logging

Tracks every action performed on patient data.

## Multi-Tenant Architecture

Supports multiple hospitals/branches.

## Security

Supports:

* Encryption
* Access control
* Compliance logging
* Data privacy

## Scalability

Supports:

* Millions of patient records
* Horizontal scaling
* Database partitioning
* High availability
